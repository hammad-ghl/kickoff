# Kickoff — Repository Intelligence Feature
## Cursor Implementation Prompt

## What We Are Building

We are adding a **Repository Intelligence** feature to Kickoff (a Design-Dev Bridge tool).

Currently Kickoff reviews a PRD in isolation — it generates test cases and checks component coverage against a UI library.

After this feature, Kickoff will also review a PRD **against the actual product codebase** — surfacing related features, hidden dependencies, and gaps the PRD doesn't address. For example: if a PRD says "add multiple reviewers to the approval flow", the system should automatically detect that:
- The approval flow also exists in the calendar view (not just the list view)
- The notification service currently only supports single-reviewer alerts
- Team permissions control reviewer assignment and may need scope changes

This is powered by a one-time vectorisation of the codebase into Qdrant, after which every PRD review gets an instant "Impact Analysis" panel.

## Architecture Overview

### The Two Phases

**Phase 1 — Indexing (one-time, runs in background, takes 5–10 minutes)**
```
User connects repo via GitHub OAuth
→ Clone/access repo via GitHub API
→ Git co-change analysis (parse git log → find files that change together)
→ Group co-changing files into feature clusters
→ Claude summarises each cluster in plain English
→ Vectorise each summary using OpenAI text-embedding-3-small
→ Store vectors + metadata in Qdrant
→ Mark repository as indexed
```

**Phase 2 — Query (runs during every PRD review, takes < 2 seconds)**
```
PRD text comes in
→ Vectorise the PRD text using same embedding model
→ Qdrant similarity search → top 6 most related feature summaries
→ Pass PRD + retrieved summaries to Claude
→ Claude outputs: related features + specific PRD gaps + what needs to be addressed
→ Display in new "Impact Analysis" panel in review UI
```

### Why This Architecture

- **No Neo4j needed** — we don't need call graph traversal. Semantic similarity search is sufficient for PRD gap detection.
- **No Tree-sitter needed** — we don't need AST parsing. Git co-change analysis gives us feature clusters without parsing code syntax.
- **Qdrant only** — one vector store, one collection called `repo_features`, handles everything.
- **All heavy AI work at index time** — query time is just a vector search + one small Claude call. This is why it feels instantaneous to the user.

## Backend Implementation

### 1. Git Co-Change Analysis Service

Create `services/gitCoChangeAnalyzer.ts`:

```typescript
/**
 * Analyses git history to find files that frequently change together.
 * Files with co-change score > threshold are grouped into feature clusters.
 * 
 * HOW IT WORKS:
 * 1. Fetch last N commits from GitHub API (no cloning needed)
 * 2. For each commit, get the list of files changed
 * 3. Build a co-occurrence matrix: for every pair of files (A, B),
 *    count how many commits touched both A and B
 * 4. Normalise: coChangeScore(A,B) = commits_touching_both / commits_touching_either
 *    (this is the Jaccard similarity coefficient)
 * 5. Group files where coChangeScore > 0.3 into clusters using
 *    a simple union-find algorithm
 * 6. Filter out clusters with only 1 file (no relationship to report)
 * 7. Filter out non-source files: ignore .md, .json, .lock, .env, config files
 */

interface CoChangeCluster {
  files: string[];
  coChangeScore: number; // average score within cluster
}

async function analyzeCoChanges(
  githubRepoFullName: string,
  branch: string,
  installationId: string,
  options: {
    maxCommits: number;      // default: 500
    minCoChangeScore: number; // default: 0.3
    minClusterSize: number;   // default: 2
  }
): Promise<CoChangeCluster[]>
```

**Implementation notes:**
- Use GitHub REST API `/repos/{owner}/{repo}/commits` to fetch commits (paginate up to maxCommits)
- Use `/repos/{owner}/{repo}/commits/{sha}` to get files changed per commit
- Rate limit: batch requests, add 100ms delay between batches of 10
- Skip merge commits (commits with more than 1 parent)
- Skip commits touching more than 50 files (they're usually dependency updates or large refactors)
- Use the existing GitHub installation token from the OAuth flow

### 2. Feature Cluster Summarisation Service

Create `services/featureClusterSummariser.ts`:

```typescript
/**
 * Takes a cluster of co-changing files and asks Claude to describe
 * what product feature those files collectively implement.
 *
 * PROMPT STRATEGY:
 * We do NOT send full file contents — too expensive, too much noise.
 * We send: file paths + file names + (if available) first 50 lines of key files.
 * File paths alone carry enormous semantic information:
 *   /modules/approval/useApprovalFlow.ts → "approval flow logic"
 *   /components/social/CalendarView.vue  → "calendar view component"
 *   /api/posts/reviewers.ts              → "reviewers API endpoint"
 */

const SUMMARISATION_PROMPT = `
You are analysing a group of files from a software product codebase.
These files frequently change together in git history, which means they
likely implement the same product feature or are tightly coupled.

File paths in this cluster:
{FILE_PATHS}

{OPTIONAL_FILE_PREVIEWS}

Based on these file paths and any code previews provided, respond with a JSON object:
{
  "feature_name": "short name for this feature (3-6 words)",
  "summary": "plain English description of what this feature does, from a product perspective. 2-4 sentences. Focus on user-facing behaviour, not implementation details.",
  "user_flows": ["list of user flows this feature supports"],
  "constraints": ["known limitations or constraints in the current implementation"],
  "dependencies": ["other features or services this depends on"],
  "affected_by": ["other features or changes that could affect this"],
  "confidence": "high | medium | low"
}

Respond ONLY with the JSON object. No preamble, no explanation.
`;
```

**Implementation notes:**
- Process clusters in batches of 5 (parallel Claude calls)
- For the first 3 files in each cluster, fetch first 50 lines from GitHub API to provide as previews
- Skip clusters where confidence is "low" — don't store them
- If Claude returns invalid JSON, retry once with a stricter prompt, then skip

### 4. Indexing Orchestrator

Create `services/repositoryIndexer.ts`:

```typescript
/**
 * FULL INDEXING PIPELINE — called when user connects a new repository
 * 
 * Steps:
 * 1. Update repository status to 'indexing'
 * 2. Run git co-change analysis → get clusters
 * 3. For each cluster → run Claude summarisation
 * 4. For each summarised cluster → generate embedding → store in Qdrant
 * 5. Save cluster records to repo_feature_clusters table
 * 6. Update repository status to 'indexed', set feature_count, set indexed_at
 * 
 * Error handling:
 * - If any step fails, set status to 'failed' with error_message
 * - Partial indexing is fine — store whatever completed successfully
 * 
 * Progress tracking:
 * - Update a progress field on the repository record after each cluster batch
 * - Frontend polls GET /api/repositories/:id every 3 seconds to show progress
 */

async function indexRepository(repositoryId: string): Promise<void>
```

**Run this as a background job** — do not make the API route wait for it. The POST /api/repositories route should:
1. Create the repository record (status: 'indexing')
2. Trigger indexRepository() asynchronously (do not await)
3. Return the repository record immediately to the frontend

  field_schema: 'keyword',

## Error Handling Requirements

- If Qdrant is unavailable, the Impact Analysis step should fail gracefully — log the error, mark the step as "skipped", do not fail the entire review
- If GitHub API rate limit is hit during indexing, pause and retry after the reset time (available in response headers)
- If Claude returns invalid JSON during summarisation, retry once then skip that cluster
- If a repo's indexing fails partway through, preserve the clusters that were already indexed — partial intelligence is better than none
- Never block the existing review pipeline (Generate Cases, Check Coverage, Map Components) if the new Impact Analysis step fails
